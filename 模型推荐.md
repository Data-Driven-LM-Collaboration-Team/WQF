# **2025.12.17** 
## 任务定义
基于GLEMOS提供的性能矩阵（300多的模型架构+400多的图数据集）
在给定一个图学习任务的前提下，直接预测一个性能最优或近似最优的 GNN 模型结构，而不在推理阶段进行任何 GNN 的训练或验证，并且推理阶段不能做RL、搜索、进化。 

**本质上是学习一个“经验函数”：根据以往大量任务–模型–性能的记录，学会在新任务上直接做决策。**

# **2025.12.25** 
## 任务定义 GNN架构推荐
GLEMOS、MetaGL 局限于有限模型集合，数量只有300-400，并且依赖性能矩阵
NAS：模型搜索空间大，耗时间耗资源 

## Pipeline
- SFT：定义模型架构空间（参考NAS-bench-graph） 输入：图数据集的自然语言描述 输出：一个完整的、符合约束的架构表示
- RL/RM:引入模型性能信号，目的是在不破坏结构合法性的前提下，让 LLM 更倾向生成 validation performance 更高的架构。

# **2026.1.15** 

**Prompt**： 给定一个图的特征，<token><token>...<token>,请帮我设计性能较好的模型来完整节点分类(链接预测)任务，要求模型按照我的模板设计: 
{dropout:..,conv...,act...} 

**token来源**:<token>1.代表图的特征 2.与LLM的语义空间对齐 

**1.代表图的特征** 

图自监督学习，对原始图作扰动，得到不同视图，原始图为锚点，不同视图为正样本，对比函数。 

**2.与LLM的语义空间对齐** 

词嵌入，把LLM的向量空间PCA后，与自监督的向量一起训练。 

**偏向性能 RL**:LLM按照模板给出若干Model后，由RM打分，反馈结果给LLM。 

# **2026.1.29** 
**怎么把graph喂给LLM**：1.graph2text 2.graph2token 3.特征图 

**怎么让LLM根据任务给出架构**：一般情况做SFT或RL:任务多，模型少。我们的情况:任务少，模型多. 

**PPO策略:**  

特征图->LLM->GNN1->打分，比best_GNN好，+1,best_GNN=GNN1 -> 重新给出GNN2 > 比best_GNN好，+1，est_GNN=GNN2...

## Proficient Graph Neural Network Design by Accumulating Knowledge on Large Language Models（WSDM 2026）
问题1：固有的知识鸿沟：当提供纯粹的语义和属性描述时，LLMS建议过度通用的GNN，无法归纳特定于数据的配置 

问题2：外部噪音：LLMS曲解了任务的相似性，将更多的权重放在了引文网络类别上，导致了“人工幻觉”和无效的知识转移。

<img width="1115" height="431" alt="image" src="https://github.com/user-attachments/assets/5004357d-3fb2-4191-ab92-6bc79bcbfdfe" />

## Beyond Model Base Retrieval: Weaving Knowledge to Master Fine-grained Neural Network Design
1.在传统的模型开发中，我们通常只知道最终结果，但缺少对“如果把某层结构改了，效果会变化多少”这种细粒度证据的记录。-构建了一个“修改-增益图” 

2.过去有效的修改经验，在新任务上不一定有效。尤其是当任务分布发生变化（即数据特征变了）时，经验的可迁移性是动态变化的，静态的相似度判断会失效。 

3.当遇到全新的、未见过的情况（OOD）时，直接检索过去的历史证据可能完全失效，因为没有现成的匹配经验。
